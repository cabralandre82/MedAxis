# Auditoria de Acessibilidade — Abril 2026

> Revisão: 2026-04-18
> Responsável: time de engenharia
> Norma-alvo: **WCAG 2.1 níveis A + AA** (AAA intencionalmente fora de escopo)
> Ferramentas: `axe-core 4.11`, `@axe-core/playwright 4.11`, `eslint-plugin-jsx-a11y`

## 1. Contexto

Este documento registra a auditoria de acessibilidade executada em abril de 2026
como parte da meta do item #25 de `docs/PENDING.md` (“Auditoria WCAG 2.1”).
O objetivo é garantir que a plataforma Clinipharma esteja livre de violações
**críticas ou sérias** em WCAG 2.1 AA, com cobertura de regressão automatizada
em CI para que futuras mudanças não recuperem débitos de a11y silenciosamente.

A auditoria **não** busca conformidade AAA — vários critérios AAA exigem
trade-offs de design (paletas limitadas, tipografia reduzida) que ainda não
foram negociados com o time de design; fica como item opcional futuro.

## 2. Metodologia

Três camadas complementares, todas integradas ao CI:

| Camada            | Arquivo                                             | Escopo                                                      | Quando roda               |
| ----------------- | --------------------------------------------------- | ----------------------------------------------------------- | ------------------------- |
| **Lint estático** | `eslint.config.mjs` (`plugin:jsx-a11y/recommended`) | JSX/TSX — regras de nomes acessíveis, labels, roles, ARIA   | `npm run lint` em todo PR |
| **Unit (DOM)**    | `tests/unit/a11y/ui-components.test.tsx`            | Renderização SSR dos blocos atômicos + fixtures de layout   | `npx vitest` em todo PR   |
| **E2E (página)**  | `tests/e2e/smoke-a11y.test.ts`                      | Páginas públicas renderizadas em Chromium, desktop + mobile | `STRICT_A11Y=1` em CI     |

### 2.1 Por que três camadas?

- **Lint** pega padrões estáticos instantaneamente no editor (baratíssimo, não
  precisa de runtime) — ex.: `<img>` sem `alt`, `<label>` sem associação.
- **Unit DOM + axe-core** roda em jsdom contra a marcação SSR dos componentes.
  Captura violações estruturais (landmarks, relationships, ARIA inválido, ordem
  de heading). **Não** capta contraste de cor porque jsdom não calcula layout
  nem pinta pixels — isso fica para o E2E.
- **E2E + `@axe-core/playwright`** roda em Chromium real com CSS aplicado, captura
  problemas que só aparecem visualmente (contraste, tap-targets, reflow).
  O modo `STRICT_A11Y=1` falha o pipeline em qualquer violação crítica/séria.

Os três dimensionam diferente: lint é instantâneo, unit é de sub-segundos, E2E
leva ~1 min por rota. Não são substituíveis — são complementares.

## 5. Correções aplicadas (resumo)

Em todas as correções os três princípios foram aplicados consistentemente:

1. **Botões icon-only recebem `aria-label` em português** descrevendo a ação
   (ex.: `aria-label="Editar categoria {nome}"`, `aria-label="Fechar diálogo"`).
   `title=` é mantido em paralelo para o tooltip visual.
2. **Ícones decorativos recebem `aria-hidden="true"`** para que o screen reader
   não leia o nome do ícone duplicadamente ao lado do texto visível.
3. **Botões sem `type=`** foram explicitamente marcados `type="button"` para
   evitar submissões acidentais em forms ancestrais.

Correções estruturais adicionais:

- `<nav>` que tinham mais de uma instância por página ganharam `aria-label`
  distinto (`"Navegação principal"`, `"Paginação"`, `"Rodapé"`).
- Link ativo na sidebar recebeu `aria-current="page"`.
- Tabela de categorias ganhou `<caption class="sr-only">` e `scope="col"` em
  todos os `<th>`.
- Modal manual em `save-template-modal.tsx` ganhou `role="dialog"`,
  `aria-modal="true"`, `aria-labelledby={title-id}`.
- Busca global ganhou `role="search"` no container, `<label class="sr-only">`
  no input, `type="search"` (em vez de `type="text"`).
- `sr-only` text em inglês foi traduzido para pt-BR em `dialog.tsx` e `sheet.tsx`.

## 6. Cobertura de regressão

### 6.1 Testes adicionados em `tests/unit/a11y/ui-components.test.tsx`

18 testes (contra o baseline de 0 — o arquivo é novo), divididos em 4 describes:

- **atomic UI components (11 testes)** — padrões canônicos de botões, forms,
  links, imagens, tabelas e hierarquia de heading. Funcionam como “fixtures
  verdes” — qualquer regressão na marcação dos componentes do design-system
  quebra o teste correspondente.
- **LegalLayout (1 teste)** — renderiza o shell usado por `/terms`, `/privacy`,
  `/dpo`, `/trust` com conteúdo realista (Section, Sub, P, UL, Highlight).
- **auth shell (1 teste)** — reproduz o form de login dentro do shell.
- **fixed regressions — lock-in (5 testes)** — reproduz a marcação corrigida dos
  componentes acima (dialog close, cursor-pagination, notification-bell, sidebar,
  save-template-modal) para impedir que uma refatoração futura silenciosamente
  volte a introduzir as violações.
- **RootLayout invariants (1 teste)** — verifica que `lang="pt-BR"`, skip-link
  e `#main` existem. Se alguém remover o skip-link, o teste pega.

```bash
# Auditoria completa (unit + lint)
npx vitest run tests/unit/a11y
npx eslint 'components/**/*.tsx' 'app/**/*.tsx'

# Auditoria E2E (precisa de Supabase creds no env)

# Contra staging real
STRICT_A11Y=1 BASE_URL=https://staging.clinipharma.com.br \
  npx playwright test smoke-a11y
```

## 7. Limitações conhecidas

- **Contraste de cor em jsdom**: axe-core não calcula pixels em jsdom, então a
  regra `color-contrast` fica desabilitada nos testes unit. É coberta pelo
  E2E — basta rodar `STRICT_A11Y=1` localmente ou em CI.
- **Componentes `@base-ui/react`**: o design-system delega grande parte da ARIA
  à `@base-ui`. Upgrade da lib pode mudar comportamento; os testes de regressão
  dependem da versão instalada. Se o upgrade for maior, re-rodar a auditoria.
- **Níveis AAA não auditados**: critério 1.4.6 (enhanced contrast), 2.4.9 (link
  purpose in isolation) etc. ficam fora. O smoke E2E usa só tags
  `wcag2a/wcag2aa/wcag21a/wcag21aa`.
- **PWA / Service Worker (item #26 de `PENDING.md`)** não faz parte desta
  auditoria; é feature separada, não requisito WCAG.

## 8. Manutenção contínua

- **Todo PR com mudança de UI** deve passar por `npm run lint` (já bloqueia) e
  `npx vitest run tests/unit/a11y` (roda junto com a suite unit completa).
- **Todo mês/release candidate** a equipe deve rodar `STRICT_A11Y=1 playwright`
  contra staging para capturar regressões visuais que passam pelo lint/unit.
- **Antes de adicionar um novo componente `<button>` icon-only**, o autor deve
  incluir `aria-label` em pt-BR e `aria-hidden="true"` no ícone — padrão
  aplicado em todo o codebase e coberto por lint.
- **Antes de adicionar um novo `<nav>`**, se já houver outro na página, adicionar
  `aria-label` descritivo.
- **Dialog customizado** (não via `components/ui/dialog.tsx`) deve ter
  `role="dialog"`, `aria-modal="true"` e `aria-labelledby` apontando para o
  heading; template: `components/orders/templates/save-template-modal.tsx`.
